# Отчёт по ручному тестированию и обзору кодовой базы (без правок кода)

Документ зафиксирован по состоянию репозитория на момент просмотра. Ниже — формулировка найденных и предполагаемых проблем, связь с текущей архитектурой (middleware, layouts, формы, логирование) и перечень файлов-кандидатов на удаление как неиспользуемых.

---

## 1. Админка: редирект и 404; один сеанс / один админ; несколько администраторов

**Наблюдение:** второй человек при попытке зайти в админку получил 404 на этапе редиректа; предположение — «токен забрал» первый пользователь, страница логина «отключается».

**Что видно по коду:**

- В `middleware.ts` защита `/admin` и `/auth/login`: при неавторизованном или не-админе запрос к `/admin` перенаправляется на `/auth/login`, с очисткой cookie `auth_token`. Страница логина middleware не отключает: для пути `/auth/login` при уже валидном JWT выполняется редирект на `/admin/orders`.
- Авторизация — JWT в httpOnly-cookie `auth_token` (см. `lib/auth.ts`, `setAuthCookie`). Срок жизни токена в JWT — до 7 дней, у cookie `maxAge` — 1 час (`60 * 60`): после истечения cookie браузер перестаёт отправлять сессию, хотя JWT формально ещё мог бы быть валидным при другой схеме — возможны расхождения ожиданий («вылетело раньше»).
- Логин ищет одного пользователя в БД по `username` (`app/api/auth/login/route.ts`, модель `User`). Поддержка **нескольких независимых админских учёток** — это отдельная продуктовая задача: в БД можно хранить несколько пользователей с `role: 'admin'`, но отдельного «лимита одного админа» в коде не видно; конфликт двух людей в **разных** браузерах обычно не из-за «одного токена на сервере».
- Ответ логина отдаёт клиенту JSON с `user.id`, `username`, `role` и дублирует `token` в теле ответа (хотя токен также ставится в cookie). Это не 404, но увеличивает поверхность утечки в DevTools/прокси.

**Возможные причины именно 404 (нужно воспроизводить с URL и окружением):**

- неверный базовый URL при редиректе (деплой, префикс, другой домен);
- запрос не на тот маршрут (опечатка `/admin` vs фактический путь приложения);
- кэш/CDN отдаёт старую сборку без маршрута `auth/login`;
- редкий сценарий с одновременным доступом из одного профиля браузера и подменой cookie.

**Требование заказчика:** явная поддержка **двух (и более) админов с разными учётными данными** и с политикой безопасности (смена дефолтных паролей, отсутствие хардкода в скриптах — см. также `create-admin.js` в разделале про мусор и секреты).

## 2. Быстрое переключение вкладок / разделов: рябь, «прыжки» вёрстки, необходимость лоадера

**Наблюдение:** при быстром переключении контент «мелькает», фон/картинка рябит; контент прижимается вверх, пока товары не загрузились — нужен явный лоадер на уровне страницы/шелла.

**Что видно по коду:**

- На главной блок «Наша коллекция» (`features/app/catalog/HomeCatalogSection.tsx`) при `loading` показывает сетку из `ShopItemSkeleton` — локальный скеллон есть.
- Каталог категорий/подкатегорий (`app/client/components/catalog/Catalog.tsx`) тоже переводит `loading` в скелетоны.
- Корневой клиентский каркас (`app/client/layout.tsx`) **не** оборачивает контент в `min-height: 100vh` / flex-колонку «хедер — растяжимый main — футер»: при пустом или низком контенте блоки визуально могут «съезжать», фиксированный фон на весь экран и контент без гарантированной минимальной высоты main усиливают ощущение дёрганья при смене страниц и при отсутствии товаров.
- Анимации страниц: `FadeWrapper` (`app/client/components/FadeWrapper.tsx`) оборачивает `children` в `AnimatePresence` с `key={pathname}` — при частой смене маршрутов возможны наслоения fade и визуальный «шум» (особенно на слабых GPU или при тяжёлых изображениях).
- Компонент `app/client/components/PageTransition.tsx` с вариантами framer-motion **нигде не импортируется** — дублирующая/мёртвая заготовка относительно текущей схемы с `FadeWrapper`.

**Итог формулировки задачи:** нужен согласованный **глобальный или секционный лоадер** (не только скелетоны сетки), стабилизация высоты области контента при загрузке и снижение визуального мерцания при агрессивной навигации (в т.ч. с точки зрения доступности — вспышки контраста).

## 3. Корзина: самовывоз — показать адрес магазина

**Наблюдение:** при выборе «Самовывоз» пользователю неудобно искать адрес на сайте.

**Что видно по коду:**

- Форма заказа (`features/app/cart/ui/OrderForm.tsx`): поле «Адрес доставки» отключается при `!isDelivery`; блок способа получения — два radio: доставка / самовывоз. **Отдельного блока с адресом точки самовывоза нет.**
- Адрес организации уже может приходить из публичных настроек (`address` в `settings`, футер и шапка получают `plainSettings` в `app/client/layout.tsx`).

**Задача:** продублировать адрес магазина (из тех же настроек, что и сайт) рядом с опцией самовывоза, чтобы не заставлять пользователя листать сайт.

## 4. Футер: прижать к «низу пола», убрать белые полосы

**Наблюдение:** под футером или между блоками появляются лишние пустые полосы (белые отступы).

**Что видно по коду:**

- `Footer` (`app/client/components/layout/Footer/index.tsx`) — обычный `motion.footer` с `py-10`, без привязки к низу вьюпорта.
- Родительский layout — фрагмент с фиксированным фоном на весь экран и последовательностью: хедер → `FadeWrapper` с контентом → плавающие соц-кнопки → футер. Нет классической схемы **flex/min-h-screen** для прижатия футера к низу окна при короткой странице.
- Корневой `app/layout.tsx`: `body` без классов растяжения на всю высоту.

**Итог:** проблема согласуется с отсутствием sticky-footer layouts; белые полосы могут быть следствием сочетания фона `html/body`, фиксированного слоя с картинкой и неунифицированных отступов у страниц.

## 5. Анимации переходов между разделами только с главной

**Наблюдение:** анимация перехода заметна при уходе с главной, при переходе между разделами «изнутри» — нет или слабо.

**Что видно по коду:**

- Глобально для витрины используется `FadeWrapper` с `key={pathname}`.
- В App Router разные ветки маршрутов оборачиваются своими экземплярами layout (`app/(root)/layout.tsx`, `app/category/layout.tsx`, `app/client/layout.tsx` для сегмента `/client/...`). При переходах между **разными деревьями layout** React может размонтировать предыдущий layout целиком: визуально один сценарий «как с главной» и другой «между разделами одного типа» могут отличаться.
- Отдельный `PageTransition.tsx` не подключён — единая стратегия переходов не доведена.

**Задача:** унифицировать переходы для **всех** клиентских маршрутов (возможно `template.tsx`, общий layout-уровень, или согласованный `AnimatePresence` с корректными `key` и режимом).

## 6. Категории на главной: настраиваемый фон блока из админки

**Наблюдение:** блок «встречающих» категорий должен иметь фон, редактируемый в админке.

**Что видно по коду:**

- Модель категории в БД (`models/Category.ts` через `createSupabaseModel`) описывает поля вроде `name`, `slug`, `subcategories` и т.д.; **отдельного поля «фон секции на главной» в типе не видно**.
- UI баннера/секций на главной (`HomeBanner`, `HomeCatalogSection`, `CategoryGrid` и т.д.) использует зашитые классы/картинки, а не данные из админки для фона всего блока категорий.

**Задача:** расширить модель настроек категории или глобальные настройки главной, админ-UI и публичный рендер.

## 7. Баннер: фон и до 6 изображений с поочерёдным показом и анимацией

**Наблюдение:** нужны настройка фона баннера и карусель/slideshow до 6 кадров с fade или каруселью.

**Что видно по коду:**

- `app/client/components/catalog/HomeBanner.tsx`: один статический `Image` с `src="/image/bannerpicture.png"`, градиент поверх; **нет** привязки к БД, **нет** слайдера.
- Настройки сайта (`models/Settings.ts`) содержат SEO, соцсети, адрес и т.д., но **нет** полей под массив изображений баннера.

**Задача:** модель + админка + клиентский компонент (таймер, fade, ограничение 6, fallback).

## 8. Инъекции (безопасность ввода и API)

**Наблюдение / запрос:** проверить устойчивость к инъекциям разных типов.

**Контекст для будущей проверки:**

- Публичные API частично открыты middleware для GET товаров/категорий/настроек и POST заказов (`middleware.ts`).
- Валидация должна быть проверена по всем полям форм (имя, телефон, email, адрес, текст заказа), query-параметрам поиска/фильтров и админским CRUD.
- Отдельно стоит проверить XSS при выводе пользовательского/админского текста в HTML, и инъекции в NoSQL/JSON, если где-то формируются небезопасные запросы к хранилищу.

Это **задача на тестирование и последующий хардненинг**, в коде изменений по этому документу нет.

## 9. Консоль разработчика: ID товаров и прочие «бекендовые» данные

**Наблюдение:** в консоли видны идентификаторы товаров; нужно скрыть всё, что раскрывает внутренности бэкенда.

**Что видно по коду:**

- В **клиентских** `features/**` и `app/client/**` массовых `console.log` для отладки не найдено; в `next.config.js` включено `compiler.removeConsole` для **production**-сборки клиента — это убирает `console.*` из бандла фронта в проде, **но не убирает** логи на сервере (Node при API routes) и не скрывает данные в **вкладке Network** (ответы JSON с `_id`, внутренние поля).
- Карточка товара получает `id={String(product._id)}` в проп `ShopItem` (`HomeCatalogSection`, `Catalog`) — в **React DevTools** дереве компонентов это всегда видно; в обычной консоли — только если что-то логирует объект или включены расширения.
- **Серверное** логирование широко использует `console.log` с сущностями БД, в т.ч.:
  - `lib/cache.ts` — категории, подкатегории, **ID** в логах;
  - `app/api/products/route.ts` — query и массив продуктов;
  - `app/api/categories/route.ts` POST — тело запроса, сохранённая категория;
  - множество других API-файлов.

Эти логи попадают в **серверные** логи хостинга (Vercel, Docker), а не в браузер пользователя; если в консоли браузера всё же видны ID — искать клиентский `console`, логирование в хуках, ошибки с прикреплённым стеком и ответами fetch, либо путать с React DevTools/Network.

**Цель:** минимизировать утечку внутренних идентификаторов и структур в **клиент** (токены, сырые документы), убрать/приглушить отладочные логи на API в продакшене, не отдавать лишние поля в публичных JSON (например, только публичный `slug` там, где ID не нужен).

## 10. Кратко о связанных рисках (не из списка заказчика, но всплыло при обзоре)

- **`create-admin.js`** при успехе пишет в консоль фиксированные логин и пароль — плохая практика и риск утечки при копировании логов.
- **Дубликаты скриптов:** корневой `generate-secrets.js` и `scripts/generate-secrets.js`; `seed-db.js` и `seed-data.js` по началу файла выглядят полными дубликатами — в `package.json` задействован только `seed-data.js` как `db:seed`.

## Кандидаты на удаление / вынос как «не используются в приложении»

Проверка по импортам и скриптам внутри репозитория (не гарантирует, что файл не нужен вам вручную вне npm).

| Файл | Почему кандидат |
|------|-----------------|
| `app/client/components/PageTransition.tsx` | Ни один `.ts`/`.tsx` не импортирует; анимации делаются через `FadeWrapper`. |
| `app/client/components/Preloader.tsx` | Нет импортов из остального кода. |
| `app/lib/mongodb.ts` | Реэкспорт `connect` из `@/lib/db`; по поиску **нет** импортов `@/app/lib` или этого пути. Вводит в заблуждение именем (Mongo), при этом `lib/db` — Supabase. |
| `components/ui/Button.tsx` | Нет импортов. |
| `components/ui/Input.tsx` | Нет импортов. |
| `components/ui/ErrorMessage.tsx` | Нет импортов из приложения (только сам файл). |
| `components/ui/Notification.tsx` | Нет импортов (уведомления админки через `react-toastify` и кастомные компоненты). |
| `image-loader.js` | Не подключён в `next.config.js` как `images.loader`; нет ссылок в проекте. |
| `lib/cors.ts` | Не импортируется маршрутами/ middleware. |
| `app/admin/products/mockProducts.ts` | Экспорт `mockProducts` нигде не импортируется. |
| `app/admin/settings/mockSettings.ts` | Аналогично, `mockSettings` не используется. |
| `seed-db.js` | Дубликат `seed-data.js`; в `package.json` скрипт `db:seed` указывает на `seed-data.js`. |
| `scripts/generate-secrets.js` | Дубликат смысла с корневым `generate-secrets.js`, который привязан к npm-скрипту `generate-secrets`. |

**Осторожно перед удалением:**

- Убедиться, что никто не запускает `node seed-db.js` или `node scripts/generate-secrets.js` в CI/документации.
- UI-компоненты из `components/ui/` могли быть заделом под дизайн-систему — перед удалением сверить с командой.
